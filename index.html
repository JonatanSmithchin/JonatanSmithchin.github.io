servlet

发表于 2022-08-10

继承关系

  1. HttpServlet->genericServlet->Servlet
  2. 核心方法:init() service() destory()
  3. service():当有请求过来,service()会自动相应 (tomcat容器调用)
    在HttpServlet中会分析请求方式,决定调用get/post/head/delet等等方法
    在HttpServlet中,这些方法默认为405风格——需要用子类实现对应方法
  4. 在新建servlet时,根据请求方式决定重写处理方法

Servlet生命周期

  1. 生命周期:从出生到死亡的过程.对应了servlet中的三个方法:init() service() destory()
  2. 默认情况下:
    1. 第一次接受请求时,此servlet会实例化(tomcat容器:内部调用构造方法)、初始化(调用init())、然后服务(调用service())
    2. 从第二次开始,直接调用服务(service())
  3. 注意:
    1. tomcat只会创建一个servelet实例,所有的请求都是由该实例去响应
    2. 默认情况下,在第一次请求时tomcat才会去实例化,初始化,以提高系统的启动速度
    3. 若需要提高响应速度,则在web.xml中设置servlet的<load-on-start>标签设置初始化时机
  4. servlet初始化时机
    1. 默认情况下是第一次接受请求时进行实例化和初始化
    2. <load-on-startup>来设置servlet启动先后顺序,数字越小,启动越靠前,最小值为0
  5. servlet在tomcat容器中是单例的、线程不安全的
    1. 单例:servlet中所有请求都是通过一个实例去相应
    2. 线程不安全:当一个线程需要根据该实例中的某个成员变量去做逻辑判断是，如果另一个线程改变成员变量，则会导致错误
    3. servlet线程不安全:
      1. 尽量避免在servlet中定义成员变量
      2. 避免根据成员变量进行逻辑判断

Docker学习

Docker概述

  • 出现
    开发–上线 两套环境 应用环境和应用配置各不相同 在自己的电脑上可以运行
    环境配置十分麻烦！部署环境费时费力,而且不能跨平台

    发布项目时，项目带上环境打包—> Docker!
    Java — jar — 打包项目带上环境（镜像） — （Docker仓库:类似于应用商店） — 下载发布的镜像 — 直接运行

    • 传统： 开发 ->jar,运维 ->部署
    • 现在： 开发打包部署上线一套流程做完。

  • Docker的思想来自集装箱

    • JRE –多个应用（端口冲突）–原来是交叉的
    • 隔离：docker的核心思想 打包装箱 每个箱子相互隔离
    • Docker通过隔离机制，可以将服务器利用到极致

  • Docker 轻巧 VM 笨重

  • 虚拟机和Docker容器技术都是属于虚拟化技术

  • VM: Linux centos原生镜像（相当于一台电脑） 如需隔离要启动多台虚拟机

  • Docker: 隔离、镜像（最核心环境+其他必要环境）小巧，只需运行镜像 几M 秒级启动

  • Docker文档 超级详细！

    Docker作用

  • 虚拟机技术缺点

    • 资源占用多
    • 冗余步骤多
    • 启动慢

  • 容器化技术

    • 容器化技术不是模拟的一个完整的操作系统

  • 不同

    • 传统虚拟机：虚拟出硬件，运行一个完整的操作系统，然后再在操作系统上安装和运行软件
    • 容器：容器内的应用直接运行在宿主机内容，容器没有自己的内核，也没有虚拟硬件，轻便
    • 每个容器见相互隔离，每个容器中都有自己的文件系统 ，互不影响

  • DevOps(开发运维)

    • 应用更快速地交付和部署
      • 传统：一堆帮助文档、安装程序
      • Docker: 打包镜像发布测试，一键运行
    • 更便捷的升级和扩缩容
      使用Docker后，部属应用就像搭积木。将项目打包为镜像，便于更快的扩展。
    • 更简单的系统运维
      容器化之后，开发、测试环境高度一致
    • 更高效的计算资源利用
      Docker是内核级别的虚拟化，可以在一台物理机上运行多个内容示例,服务器的性能可以被压榨到极致！

Docker架构

  • 镜像(image):
    docker类似模板，可通过镜像(模板)来创建容器服务，可创建多个容器。(最终服务或者项目运行在容器中)
  • 容器(container):
    利用容器技术，通过镜像来创建，运行一个或者一个组应用。
    可暂时理解为简易的linux系统
  • 仓库(repository):
    存储镜像的地方。(公有仓库和私有仓库)
    Docker Hub、阿里云等

    底层原理

    Docker怎样工作？

    Docker是一个Client-Server结构的系统，Docker的守护进程运行在主机上。通过socket从客户端访问。DockerServer接收到Docker-Client指令，就会执行。

    Docker为什么比VM快？

  1. Docker有比虚拟机更少的抽象层
  2. Docker利用的是宿主机的内核，VM需要重新加载操作系统。Docker省略了复杂的操作过程。

用户登录

  • 使用ObjectIOStream在客户端和服务器间传输User和Message对象。

  • 验证用户名是否正确

    • 客户端:向服务器发送User对象,再接受服务器发回的Message对象。根据Message的MessageType属性判断是否登陆成功。
      • 登陆成功,则在客户端创建并启动一个新的含有Socket的线程,持续监听来自服务器的Message对象。
      • 登陆不成功,则关闭Socket重新尝试登录。
    • 服务器:通过一个端口持续监听来自客户端的Socket。若接收到Socket,则接收User对象并开始验证。
      • 若验证通过,则返回MessageType==MESSAGE_LOGIN_SUCCEED的Message对象,创建并启动一个新的含有Socket的线程,持续监听相应客户端的线程。
      • 若验证不通过,则返回MessageType==MESSAGE_LOGIN_FAILED的Message对象并关闭Socket。

  • 使用 HashMap 管理建立起的多线程。

  • Socket是每个线程中的属性,各不相同。客户端 每个线程中的Socket都和 服务器 中一个线程的Socket建立了联系。

  • 在线程while循环持续监听的过程中,若接收不到Message,则该线程会 阻塞 ,直到接收到Socket。

  • 客户端仅仅退出了main方法中的while循环,却没有停止与服务器通讯的线程？服务端异常奔涌而出,卡到IDEA行动不能? 赶快实现客户端无异常退出！

网络编程

网络通信协议

  • TCP Transmission Control Protocol
    • 面向连接的通信协议,连接后要进行 “三次握手”确认连接 可靠 。
    • 三次握手后,Client与Server间可进行大量数据的传输。虽然效率较低，但可保证传输的安全。
  • UDP User Datagram Protocol
    • 消耗资源少,传输效率较高,传输文件有大小限制（64K）。
    • 不能保证数据传输的完整性。

UDB网络通信编程

  • 没有明确的服务端和客户端，演变成数据的发送端和接收端
  • 接收和发送数据通过DatagramSocket对象完成
  • 将数据封装到DatagramPacket对象（装包）—>发送—>接受到DatagramPacket后拆包取出数据
  • DatagramSocket可指定端口接收数据

    基本流程

  • 可以看出接收端和发送端基本是 平等 的关系,两者没有显著区别。
